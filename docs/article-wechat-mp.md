# 我用微信接管了 Claude Code：一个 1072 行的 AI Agent 实验

---

三月的一个晚上，我在地铁上突然想到一个 bug 的修法。

掏出手机，打开微信，给自己的 Bot 发了一条消息：

> 帮我看看 src/auth.ts 里的 token 校验逻辑，第 42 行附近好像有个边界条件没处理。

四十秒后，Bot 回复：

> 找到了。第 43 行的 `if (token.exp > Date.now())` 应该用 `>=`，当 token 恰好在当前毫秒过期时会被误判为有效。已修复并保存。

**它不是在"聊天"。它真的改了代码。**

这就是我做 weixin-claude-bot 的起因——用微信消息远程操控 Claude Code，让 AI 帮你读代码、改代码、跑命令。

---

## 两个东西碰在了一起

这个项目能成立，靠的是两个时机恰好碰在一起的东西。

**第一个：腾讯悄悄开放了微信 Bot 协议。**

2026 年 3 月，一个叫 `@tencent-weixin/openclaw-weixin` 的 npm 包出现了。3756 行完全可读的 TypeScript，MIT 开源，没有公告，没有文档。

它做了一件以前不可能的事：让第三方程序通过标准 HTTP 和微信收发消息。

微信一直是封闭生态的代名词。之前想接入微信，只能靠灰色手段——网页版协议逆向、客户端 Hook 注入——随时面临封禁。而这个包的出现，像是腾讯在 AI Agent 时代悄悄推开了一扇门。

这套协议叫 **iLink**，设计极简：5 个 HTTP 端点，覆盖了即时通讯的全部核心功能。

**第二个：Anthropic 开放了 Claude Code 的 SDK。**

Claude Code 很多人用过——在终端里跟 AI 对话，它能帮你读写文件、执行命令、搜索代码库。

但大部分人不知道的是，这个能力可以作为 SDK 嵌入到你自己的程序里。一个函数调用，就能启动一个完整的 Claude Code 实例。

当这两个东西碰在一起，一个自然的想法产生了：

**把微信当作遥控器，把 Claude Code 当作执行器。**

---

## 它不是聊天机器人

这里需要强调一个区别，因为很多人会搞混。

普通的 AI 聊天机器人是这样的：

> 你发一段文字 → AI 想一想 → 回你一段文字

Claude Agent SDK 是这样的：

> 你发一段指令 → AI 启动一个子进程 → 读文件、执行命令、搜索代码 → 自主循环多轮 → 把结果告诉你

区别在于 **"手"**。普通聊天机器人只有嘴，Claude Agent 有手——它内置了 20 多个工具，能真正操作你的开发环境。

当你说「帮我把所有 var 改成 const」，它不是告诉你怎么改，而是真的去搜索、逐个打开文件、修改、保存。

---

## 架构：简单到无聊

整个数据流只有一条链：

```
微信 → iLink 服务器 → Bot (Node.js) → Claude Code 子进程 → 本地文件
  ←                  ←              ←                     ←
```

Bot 做的事情很朴素：

1. 长轮询 iLink 服务器，等微信消息
2. 收到消息，提取文本
3. 发一个「正在输入...」状态（微信里那个跳动的省略号）
4. 把文本丢给 Claude Agent SDK
5. 等 Claude 做完所有事情，拿到最终结果
6. 发回微信

没有消息队列，没有数据库，没有缓存层。所有状态存在本地 JSON 文件里。

**整个项目 1072 行 TypeScript，零框架依赖。**

## 逆向 iLink：从 3756 行到 200 行

iLink 没有文档。理解它的唯一方式是读 `openclaw-weixin` 的源码。

但这个包和 OpenClaw 框架深度耦合——它 import 了一堆 `openclaw/plugin-sdk/*` 的内部模块，没法直接在独立项目里用。

所以我们做了一件事：读懂协议，用最少的代码重写。

**从 3756 行到 200 行。**

这个过程中有几个有意思的发现。

**发现 1：context_token 是回复的"钥匙"**

每条收到的消息带一个 `context_token`。回复时必须原样带上，否则消息发不出去。这个设计意味着 Bot 只能回复，不能主动找人说话。安全，但也有局限。

**发现 2：from_user_id 必须留空**

直觉上，发消息应该填上自己的 ID。但 iLink 的设计是服务端根据认证信息自动填充。你自己填了反而会出问题——表现为第一条消息正常，之后全部消失。这种 bug 极难排查。

**发现 3：每条消息需要唯一 client_id**

没有它，服务端会把后续消息当作第一条的重复，静默丢弃。两个 bug 叠在一起，让「只有第一条回复正常」成了最经典的症状。

## Claude Agent SDK：76 行集成一个 AI Agent

这是整个项目最核心的文件，只有 76 行。

它只做一件事：调用 `query()` 函数，等 Claude 完成所有操作，返回最终文本。

SDK 的 `query()` 返回一个消息流。Claude 在工作过程中会产生很多中间消息——「让我搜索一下...」「找到了这个文件...」「正在修改...」——这些对微信用户没有价值。

我们的处理方式：收集所有中间文本，但当最终结果到达时，**全部清空，只保留最终总结**。

这个设计决策很重要。微信不是终端，用户不想看到 AI 的思考过程，只想要结果。

---

## 多轮对话：记住你上一句说了什么

默认情况下，每条微信消息是独立的 Claude 会话。你问完 A 问题再问 B 问题，Claude 完全不记得 A。

我们用 SDK 的 `resume` 机制解决了这个问题：每次调用结束后保存 session ID，下次调用时传入，Claude 自动恢复完整上下文。

开启方式：`npm run config -- --multi-turn true`

发送「新对话」「/reset」或「/clear」可以重置，开始全新对话。

这里有一个工程上的细节：如果 resume 失败（比如 session 过期），Bot 不会卡死，而是自动清除这个用户的 session ID。下一条消息会自动开始新对话，用户甚至不会感知到异常。

---

## 安全：一个开放的问题

有一个问题必须正视：**通过微信发来的任何文本都会被当作指令执行。**

如果有人给你的 Bot 发了「删除所有文件」，Claude 真的可能去执行。

SDK 提供了几种防护机制：

- **auto 模式**：一个 AI 分类器在每次执行命令前判断是否安全，会拦截 `rm -rf`、`git push --force` 这类操作
- **cwd 限制**：把 Claude 的工作目录限定在特定项目里，不让它访问其他地方
- **canUseTool 回调**：最精细的控制——自定义函数逐个审查每次工具调用

目前项目默认用 `bypassPermissions`（全放开），因为 `auto` 需要 Team 订阅。这是一个有意的取舍：作为实验教学项目，简单优先。

但如果你要在实际场景使用，**权限是必须认真对待的**。

## 1072 行代码的启示

这个项目让我重新思考了一个问题：在 AI Agent 时代，程序员的核心能力是什么？

不是从零实现一个 LLM。不是自己写文件操作和代码搜索工具。不是造一个即时通讯协议。

而是 **理解已有的能力，并用最少的代码把它们组合在一起**。

iLink 给了我们微信消息的通道。Claude Agent SDK 给了我们 AI Agent 的执行力。我们写了 1072 行 TypeScript 做了一根管子，把两端连起来。

这根管子很简单。但它让你能在地铁上改代码。
